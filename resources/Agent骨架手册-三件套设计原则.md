# Agent 骨架手册：SOUL · TOOLS · MEMORY 的设计原则

> 作者：小敏 | 2026-04-15 | 来源：飞书文档

## 一句话总览

一个 agent 能不能长期用，不取决于模型多强，而取决于它有没有稳定的"三件套"：灵魂文件（SOUL）、工具手册（TOOLS）、记忆系统（MEMORY）。

## 一、三件套的分工

| 文件 | 作用 | 谁改 | 多久读一次 |
|------|------|------|-----------|
| SOUL.md | 人格 + 工作原则 + 绝不违反的铁律 | 人和AI共同演进，变动慢 | 每次会话启动必读 |
| TOOLS.md | 具体命令、脚本、API、踩过的坑 | 主要是AI维护，人审阅 | 用到对应工具才读 |
| MEMORY.md + memory/ | 长期事实 + 当日运行记录 | 由AI持续写入 | 相关话题按需读 |

## 二、SOUL.md 结构（100行以内）

推荐结构：
- 人格（5-8条，每条一行）
- 工作原则（5-10条）
- 执行规则（最小闭环）
- 醒来先校准（3类）
- 铁律（绝不违反，5-8条）
- 纠错即沉淀
- 对外动作边界
- 文档瘦身原则

**铁律要点**：每条铁律背后都有一次真实事故。写SOUL前先问：这条规则对应哪次踩的坑？

**瘦身原则**：SOUL只能有人格/原则/铁律/入口指针，具体怎么做都外挂。

## 三、TOOLS.md 结构

```
## 主要工具 1
### 常用命令
### 关键任务ID清单
### 踩过的坑（重要！）
```

每次出事故都在对应小节加一条记录。

## 四、启动协议（AGENTS.md）

```
## 每次会话启动
1. 先读 SOUL.md
2. 再读 USER.md
3. 读 memory/YYYY-MM-DD.md
4. 主会话才读 MEMORY.md
```

## 五、从零搭架子的步骤

1. 先想清人格
2. 写 SOUL.md（人格 + 5条原则 + 3条铁律）
3. 写 AGENTS.md（启动协议）
4. 写 USER.md
5. 建 memory/ 目录
6. 先不写 TOOLS.md（用着再记）
7. 铁律后补（踩坑沉淀）

## 六、常见陷阱

1. SOUL越写越长 → 外挂到TOOLS或data
2. MEMORY成了流水账 → 按天归档
3. 自动化层和声明层脱节 → 停用必须全链路审计
4. 采集任务套LLM → 纯API拉数据用脚本，不用LLM
5. 被问状态时凭记忆回答 → 强制走实时源

## 七、工具选择原则

| 任务类型 | 用什么 | 为什么 |
|----------|--------|--------|
| API拉数据→写入库 | Python脚本+launchd/crontab | 稳定可调试 |
| 判断/推理/写报告 | LLM（好模型） | 需要语境理解 |
| 晨扫/晚扫/周报 | LLM（opus级） | 多步骤+写作质量 |

## 八、最后的忠告

三件套不是一次写完的，是被问题一次次逼出来的。

人格不对→加SOUL原则 | 踩坑→加TOOLS踩坑区 | 忘事→升级memory规则 | 凭记忆答话→加铁律
